# PQ-BitEdu 项目规划

## 1. 项目定位

### 1.1 一句话定位
这是一个**教学型、可实验、可攻防分析的类比特币后量子区块链系统**，核心卖点不是“真的发币创业”，而是：

1. 保留 Bitcoin-like 的核心机制；
2. 用 **ML-DSA** 体现“抗量子”签名层；
3. 用 **SOIBS** 体现你自己的系统级小创新；
4. 用 **多节点模拟 + 攻击实验 + Agent 经济行为** 把课程知识讲活。

### 1.2 对外表达建议
上台时建议始终把它表述为：

`教学实验链 / 课程原型 / 仿真系统`

而不是：

`真实可公开运营的新币`

这样更稳，也更容易得到老师认可，因为重点会落在“你理解了机制并做了系统设计”，而不是“你想真的搞一个币圈项目”。

## 2. 我对你当前思路的判断

你的总体方向是对的，而且有明显“比普通课程作业更卷”的潜力，但需要做一次**范围重构**。

### 2.1 值得保留的部分
以下部分很强，应该作为项目主线：

1. 先讲比特币安全来自哪些层：
   签名授权、哈希链、Merkle Root、PoW、最长链。
2. 再讲量子计算会威胁什么：
   主要威胁椭圆曲线签名，不是自动威胁所有区块链机制。
3. 再介绍主流方案：
   Bitcoin 重签名与 PoW，Ethereum 偏账户模型和智能合约，后量子方向常见签名候选。
4. 然后引出你的设计：
   用 ML-DSA 替代传统签名，并提出 SOIBS 缓解见证膨胀。
5. 最后做实验：
   交易合法性、体积对比、51% 攻击、Agent 市场模拟。

### 2.2 需要收敛的部分
以下部分如果不收敛，项目很容易失控：

1. “自己发明一个类似比特币的东西”可以做，但不要发明新的底层密码学。
   你现在文档里的做法是对的：**底层密码学使用标准 ML-DSA，创新放在系统结构 SOIBS 上。**
2. “试运营一下”建议改成：
   **封闭环境下的仿真运行**
   包括本地多节点、自动挖矿、自动交易、市场价格波动。
3. “让 Agent 炒到多少元”可以做，但要明确：
   这是**经济模拟层**，不是区块链安全性的直接证明。
4. “抗量子”要讲得克制：
   你这个系统是**签名层抗量子**，不是整个系统所有层都抗量子。

## 3. 推荐的总方案

建议把整个项目拆成三个层次。

### 3.1 第一层：核心链
这是最重要的答辩主体，必须保证能跑、能解释、能演示。

包含：

1. UTXO 模型
2. 交易输入输出
3. `pubkey_hash` 锁定
4. ML-DSA 签名与验签
5. SOIBS 授权打包
6. 区块结构
7. Merkle Root
8. PoW 挖矿
9. 最长链规则
10. coinbase 与手续费

### 3.2 第二层：安全与性能实验层
这是你和普通作业拉开差距的地方。

包含：

1. 篡改交易后签名失效实验
2. 错公钥/伪签名验证失败实验
3. SOIBS 与逐输入见证的体积对比
4. SOIBS 与逐输入见证的验签次数对比
5. 51% 攻击或双花模拟
6. 可选的自私挖矿简化实验

### 3.3 第三层：Agent 经济模拟层
这是项目的“演示亮点层”，但不应该抢核心链的主角位置。

包含：

1. Agent 拥有钱包、余额、算力、生活成本、电力成本
2. Agent 可以做三类行为：
   挖矿、转账、持币/抛售
3. 引入简单市场价格机制：
   价格由最近若干轮净买入和净卖出决定
4. 观察币价、财富分布、矿工收益和链上活跃度
5. 在攻击场景下观察市场信心和价格变化

## 4. 你这个项目最合理的“原创点”定位

最好的说法不是“我发明了一种新的加密货币算法”，而是：

**我设计了一个面向教学的后量子 Bitcoin-like 区块链系统，并针对 ML-DSA 引入后的见证膨胀问题，提出了 SOIBS 这一交易级授权打包机制。**

这个说法有几个优点：

1. 学术上更稳，不容易被追问“你是不是重新发明密码学”；
2. 工程上更真实，因为你确实是在做系统设计；
3. 展示上更高级，因为它不是简单“换个签名库”。

## 5. 实现路线建议

建议按“先稳后炫”的顺序做，不要一开始就冲 Agent。

### 5.1 Phase A：MVP 核心链
目标：先做出一个最小可运行的教学链。

必须完成：

1. 哈希与序列化
2. 交易对象与 txid
3. UTXO 集维护
4. 普通交易校验
5. 区块对象与 Merkle Root
6. PoW 挖矿
7. 最长链规则
8. coinbase 奖励

这一阶段可以先临时用假签名接口占位，先把链跑通。

### 5.2 Phase B：接入 ML-DSA
目标：把“抗量子”真正落地。

完成：

1. `SignatureScheme` 抽象
2. `MLDSASignature` 实现
3. 钱包生成密钥对
4. 交易签名与验签
5. 地址使用 `SHA256(public_key)`

### 5.3 Phase C：实现 SOIBS
目标：把你的原创点真正做出来。

完成：

1. 定义 `AuthBundle`
2. 将输入与 bundle 建立映射
3. 构造 bundle 签名消息
4. 验证 bundle 完整覆盖性
5. 比较逐输入见证和 SOIBS 见证大小

### 5.4 Phase D：多节点与攻击实验
目标：把“安全不是一句话”讲出来。

完成：

1. 单机多节点模拟
2. 节点本地 mempool
3. 矿工节点
4. 攻击者节点
5. 双花与私链重组实验

### 5.5 Phase E：Agent 市场模拟
目标：把展示效果拉满。

完成：

1. 多个 Agent 钱包初始化
2. 每轮随机或策略性生成交易
3. 每轮扣除生活成本/电费
4. 根据供需变化更新参考价格
5. 输出价格曲线和财富变化图

## 6. Agent 部分应该怎么做才不失控

你想做 Agent 是好想法，但建议做成**规则驱动 Agent**，而不是一上来接大模型。

### 6.1 第一版建议
每个 Agent 只需要这些状态：

1. `cash`
2. `coin_balance`
3. `hash_power`
4. `electricity_cost`
5. `living_cost`
6. `risk_preference`

### 6.2 行为规则建议
每轮让 Agent 在以下动作中选一个：

1. 挖矿
2. 买币
3. 卖币
4. 转账
5. 持有不动

### 6.3 定价建议
不要上复杂 AMM，课程项目里用一个简单价格函数就够：

`price_next = price_now * (1 + alpha * net_demand - beta * panic_factor)`

其中：

1. `net_demand` 表示这一轮买卖差额
2. `panic_factor` 可在攻击成功、回滚发生时上升

这样你就能讲：

1. 电力成本如何影响卖压
2. 挖矿奖励如何影响供给
3. 攻击事件如何打击市场信心

## 7. PPT 推荐结构

### 7.1 开场
问题导向：

1. 比特币为什么能在没有中心机构的情况下记账？
2. 它到底安全在哪？
3. 如果量子计算来了，哪一层先出问题？

### 7.2 基础知识回顾

1. 哈希
2. 数字签名
3. UTXO
4. Merkle Root
5. PoW
6. 最长链

### 7.3 主流链/主流币快速对照

1. Bitcoin
2. Ethereum
3. 其他后量子尝试或研究方向

重点不是讲币圈生态，而是讲：

`它们的账本模型、签名方案、共识机制有什么区别`

### 7.4 你的系统设计

1. PQ-BitEdu 目标
2. 为什么选择 ML-DSA
3. 为什么需要 SOIBS
4. 系统架构图
5. 交易结构图
6. 验签流程图

### 7.5 实验结果

1. 合法交易示例
2. 篡改交易失败示例
3. SOIBS 节省了多少体积
4. 攻击实验效果
5. Agent 市场价格曲线

### 7.6 结论

1. 抗量子只解决签名层问题
2. PoW 安全边界仍然存在
3. 系统优化可以缓解后量子签名带来的工程成本

## 8. 风险点与避坑建议

### 8.1 最大风险不是代码，而是范围
如果你同时想做：

1. 完整链
2. 真后量子签名
3. 多节点网络
4. 攻击实验
5. LLM Agent 炒币
6. 好看的 PPT

那很容易每一部分都只做了 60 分。

### 8.2 推荐优先级
真正的优先级应该是：

1. 核心链能跑
2. ML-DSA 能签能验
3. SOIBS 能证明有效
4. 攻击实验能演示
5. Agent 模拟是加分项

### 8.3 一个很重要的边界
“抗量子”不能被表述成：

`这个链因此整体绝对安全`

更准确的说法是：

`这个链在交易授权层使用了后量子签名，因此相对于传统椭圆曲线签名，更能抵抗未来量子计算对签名层的威胁。`
